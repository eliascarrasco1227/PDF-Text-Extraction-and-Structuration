
\chapter{Diseño e implementación del sistema}

En este capítulo se describe la arquitectura técnica del sistema desarrollado, detallando la lógica de cada módulo y los mecanismos de integración entre el procesamiento de documentos, la inteligencia artificial y la generación de documentación científica.

\section{Arquitectura general del sistema}

El sistema se ha diseñado bajo una arquitectura modular y desacoplada, organizada en un \textit{pipeline} de procesamiento secuencial con capas de validación. Esta estructura permite que cada componente cumpla una función única, facilitando el mantenimiento y la posible sustitución de módulos (como el cambio de modelo de IA) sin afectar al resto del ecosistema.

\subsection{Flujo de trabajo del pipeline}

El proceso de transformación de una gramática analógica en un documento estructurado sigue un flujo de cinco etapas, representadas de forma secuencial en la Figura~\ref{fig:uml_pipeline_principal}:
\begin{enumerate}
    \item \textbf{Capa de ingesta y preprocesamiento:} El sistema localiza los archivos PDF en el directorio \texttt{data/} y extrae el rango de páginas solicitado. En la ruta experimental principal, esta extracción se realiza mediante segmentación del PDF en un flujo binario (\textit{bytes}), sin rasterizar las páginas. De forma complementaria, el sistema contempla una ruta alternativa de prototipado basada en imágenes rasterizadas.
    
    \item \textbf{Capa de inferencia (Motor de IA):} Se envía el fragmento documental obtenido, principalmente como PDF segmentado en bytes, junto con un \textit{prompt} especializado y ejemplos de entrenamiento (\textit{few-shot}) para obtener una respuesta en formato XML.
    
    \item \textbf{Capa de post-procesamiento y corrección:} La salida de la IA se somete a una limpieza automática para corregir errores sintácticos comunes en la generación estocástica (etiquetas mal formadas o duplicadas).
    
    \item \textbf{Capa de validación estructural:} Se contrasta el XML generado contra el archivo de definición \texttt{gramatica.dtd} para asegurar el cumplimiento de la jerarquía lingüística.
    
    \item \textbf{Capa de exportación y tipografía:} Una vez validado, el contenido se traslada a formatos finales: XML para almacenamiento de datos, \LaTeX{} para publicación académica y PDF para mayor visibilidad.
\end{enumerate}

La Figura~\ref{fig:uml_pipeline_principal} representa el flujo secuencial del \textit{pipeline} principal, desde la petición inicial del usuario hasta la generación del XML, su validación, corrección, conversión a \LaTeX{} y compilación final a PDF.

\subsection{Organización de los módulos del sistema}

La implementación se refleja en la siguiente organización de directorios en el repositorio, representada visualmente en la Figura~\ref{fig:arquitectura_modular_repositorio}:

\begin{itemize}
    \item \textbf{\texttt{core/}:} Contiene la lógica de negocio principal, incluyendo la interacción con la IA, la gestión de archivos y el corrector de XML.
    
    \item \textbf{\texttt{config/}:} Centraliza los parámetros del sistema, rutas de directorios y la gestión de la arquitectura técnica.
    
    \item \textbf{\texttt{validator/}:} Aloja las reglas formales (DTD) y el script de validación que garantiza la calidad de los datos.
    
    \item \textbf{\texttt{latex\_generator/}:} Módulo especializado en la conversión de etiquetas XML a comandos de tipografía científica.
    
    \item \textbf{\texttt{evaluator/}:} Infraestructura para la comparación de resultados y cálculo de métricas de precisión frente al \textit{test set}.
\end{itemize}

La Figura~\ref{fig:arquitectura_modular_repositorio} muestra la organización modular del repositorio y la relación entre el orquestador principal, los paquetes funcionales y los módulos Python que intervienen en el proceso.

Esta arquitectura garantiza la trazabilidad absoluta de los datos: desde el PDF original hasta el archivo \texttt{.tex} final, cada paso es registrado y puede ser auditado mediante el sistema de \textit{logs} centralizado.

Además de las carpetas anteriores, también existen directorios destinados al almacenamiento de datos y recursos auxiliares:

\begin{itemize}
    \item \textbf{\texttt{data/}:} Contiene los datos de entrada al programa, como gramáticas en PDF, incluyendo los ejemplos de entrada y salida utilizados en el proceso de \textit{few-shot learning}.
    
    \item \textbf{\texttt{logs/}:} Almacena de manera permanente todos los archivos de texto con registros de la aplicación para poder consultar los procesos que se han ejecutado en cada momento.
    
    \item \textbf{\texttt{output/}:} Carpeta por defecto para la salida del sistema y donde se almacenan las ejecuciones anteriores para futuros análisis y comparativas.
    
    \item \textbf{\texttt{prompts/}:} Aloja los archivos de texto con las diferentes versiones de los \textit{prompts} que se han ido utilizando a lo largo del desarrollo.
    
    \item \textbf{\texttt{test\_set/}:} Centraliza los conjuntos de pruebas empleados para la evaluación de la salida del programa.
\end{itemize}

\begin{itemize}
    \item Acoplamiento visual: las etiquetas como \texttt{<columna1>} o \texttt{<fila1>} forzaban al modelo de IA a realizar un análisis geométrico del soporte físico en lugar de una interpretación lingüística, lo que aumentaba la tasa de error por la distorsión espacial de las fuentes escaneadas.

\end{itemize}

Asimismo, se optimizó la flexibilidad de la estructura eliminando contenedores intermedios rígidos para secciones complejas, lo que dotó al \textit{pipeline} de una mayor resiliencia ante la naturaleza estocástica del LLM. \subsection{Definición Formal de la Gramática de Datos (DTD)}

El contrato estructural definitivo que rige la validación sintáctica del sistema se encuentra formalizado en el archivo \texttt{gramatica.dtd}, cuyo código se detalla en el Código~\ref{code:dtd} y cuya jerarquía general se resume en la Figura~\ref{fig:esquema_xml_semantico}.

Con este archivo DTD, el documento XML resultante se articula como un árbol jerárquico de herencia y adyacencia donde se diferencian dos grandes tipologías de elementos:

\begin{itemize}
    \item Elementos de estructura editorial 
    
    (\texttt{document}, \texttt{page}, \texttt{heading}, \texttt{title}, \texttt{line}): \texttt{document} opera como el nodo raíz del sistema. Este contiene de forma iterativa uno o más elementos \texttt{page}. Cada página está dotada de un atributo obligatorio (\texttt{number}) para preservar la trazabilidad bibliográfica respecto al PDF original. El contenido interno de \texttt{page} es de tipo mixto y libre de contenedores rígidos; permite la aparición libre e intercalada de líneas ordinarias (\texttt{line}), títulos capitulares (\texttt{title}) y encabezados (\texttt{heading}), mitigando los bloqueos en el analizador sintáctico (\textit{parser}) cuando la IA procesa estructuras transicionales de maquetación.

    \item Elementos de análisis lingüístico avanzado 
    
    (\texttt{interlinear\_gloss}, \texttt{parallel\_phrase}, \texttt{morpheme\_analysis}, \texttt{syntactic\_analysis}, \texttt{unit}): el elemento central de la digitalización semántica es \texttt{interlinear\_gloss}. Como se especifica en la lógica relacional de la DTD, este nodo actúa como un orquestador que puede integrar, de manera opcional y secuencial, un bloque de traducción directa (\texttt{parallel\_phrase}), una capa de desagregación morfológica (\texttt{morpheme\_analysis}) y una capa de segmentación de sintagmas y funciones gramaticales (\texttt{syntactic\_analysis}). Las capas morfológica y sintáctica se componen obligatoriamente de una o más estructuras constitutivas denominadas \texttt{unit}. A nivel atómico, cada \texttt{unit} asocia unívocamente una forma morfológica o sintáctica concreta de la lengua maya (\texttt{form}) con su glosa explicativa parametrizada en castellano (\texttt{gloss}).
\end{itemize}

Mediante esta arquitectura relacional, el sistema busca preservar el texto plano de las gramáticas normativas y generar un mapa posicional y gramatical explícito altamente estructurado. No obstante, el grado de fidelidad alcanzado depende de la complejidad de cada página y debe evaluarse empíricamente mediante las métricas presentadas en el capítulo de resultados.



\section{Módulo de procesamiento de documentos (\texttt{pdf\_processor.py})}

El primer eslabón del \textit{pipeline} es el módulo encargado de la ingesta y manipulación de los archivos fuente. Dado que las gramáticas de la ALMG presentan una naturaleza heterogénea, se ha implementado una estrategia principal de segmentación y extracción de flujo binario que permite aislar fragmentos del documento original sin alterar su estructura interna. Esta es la ruta utilizada en la experimentación final; la conversión a imagen queda reservada como alternativa de prototipado para casos concretos en los que se requiera inspección visual aislada.

\subsection{Lógica de extracción y segmentación de bytes}

El script \texttt{pdf\_processor.py} actúa como un gestor de \textit{streams} de datos que opera directamente sobre la estructura del formato PDF. En la ruta principal del sistema, no realiza una rasterización de páginas; su función es segmentar el documento para extraer rangos específicos de páginas y servirlos en formato binario (\textit{bytes}) de forma eficiente.

\begin{itemize}
    \item Herramientas técnicas: se utiliza la librería \texttt{PyPDF2}, que permite realizar operaciones de lectura y escritura a bajo nivel sobre archivos PDF.

    \item Procedimiento: el módulo abre el archivo fuente mediante \texttt{PdfReader}, calcula el índice de las páginas solicitado realizando el ajuste de base 1 a base 0 requerido por la librería, y emplea \texttt{PdfWriter} para encapsular únicamente las páginas seleccionadas. El resultado no es una imagen, sino un objeto serializado en memoria (\texttt{io.BytesIO}) que contiene el PDF resultante, el cual se retorna como un flujo de \textit{bytes}. Este enfoque asegura que la IA reciba la información más pura posible del documento, conservando su estructura vectorial nativa.
\end{itemize}

\subsection{Gestión de la integridad y fidelidad del documento}

Un factor crítico en este desarrollo es la preservación de la integridad del documento original durante el filtrado. Al trabajar en la ruta experimental principal con la manipulación de \textit{bytes} en lugar de conversiones a imagen, el sistema garantiza:

\begin{itemize}
    \item Ausencia de degradación: al evitar procesos de binarización o compresión de imágenes, el texto y los elementos gráficos del PDF mantienen su definición nativa, facilitando que el modelo multimodal interprete caracteres pequeños y estructuras complejas con mayor precisión.

    \item Eficacia en el entorno de producción: la entrega del contenido en formato de flujo binario permite una integración directa con los \textit{endpoints} de la API de Gemini que aceptan archivos PDF como fuente, aprovechando la capacidad del modelo para ``leer'' la estructura interna del documento sin necesidad de procesar metadatos innecesarios o ruido visual generado por conversiones de formato.
\end{itemize}

\subsection{Selección de rangos y modularidad}

Para optimizar el consumo de \textit{tokens} y asegurar el cumplimiento de las cuotas diarias de la API, el procesador incluye una lógica de extracción selectiva basada en rangos. Esta funcionalidad ha sido determinante para trabajar sobre el \textit{Test Set} de forma aislada, permitiendo ejecutar pruebas rápidas, precisas y fácilmente reproducibles sin cargar volúmenes masivos de datos irrelevantes.

En resumen, \texttt{pdf\_processor.py} garantiza que el sistema gestione la ingesta de documentos de manera modular y técnica, proporcionando a la Inteligencia Artificial una entrada de datos fiel al original y optimizada para el \textit{pipeline} de digitalización estructural.





\section{Motor de generación mediante IA (\texttt{ai\_generator.py})}

Este módulo constituye el núcleo cognitivo del sistema. Su función es orquestar la comunicación con los Modelos de Lenguaje de Gran Tamaño (LLMs) de la familia Gemini, transformando los fragmentos documentales procesados en datos estructurados mediante técnicas avanzadas de ingeniería de \textit{prompts}.

\subsection{Configuración y parametrización del modelo}

El script \texttt{ai\_generator.py} gestiona la forma en la que se realiza la llamada a la API del modelo generativo. La configuración de la petición se centra en tres parámetros fundamentales:

\begin{itemize}
    \item Temperatura: parámetro central de la experimentación de este TFM, que regula el equilibrio entre determinismo y creatividad. Se ha implementado de forma variable para permitir el estudio sistemático de su impacto en la precisión estructural.

    \item Modelo: identifica el modelo utilizado para generar la respuesta. Esta constante se encuentra definida en el archivo \texttt{properties.py}.

    \item Contenido: engloba tanto el \textit{prompt} principal como los ejemplos de \textit{few-shot learning}, incluyendo sus respectivas entradas, salidas y explicaciones.
\end{itemize}

\subsection{Estrategia de inferencia: Few-Shot Learning}

Para garantizar que el modelo genere una estructura XML coherente con la gramática formal definida por la DTD, el sistema no utiliza una estrategia de \textit{Zero-Shot Learning}, sino una aproximación basada en \textit{Few-Shot Learning}.

A través del módulo \texttt{prompt\_reader.py}, el sistema incorpora a la petición ejemplos reales formados por pares de entrada y salida compuestos por una imagen y su correspondiente XML validado, almacenados en la carpeta \texttt{data/few-shot}.

Esta técnica proporciona al modelo un patrón visual y sintáctico previo, reduciendo significativamente las alucinaciones estructurales y favoreciendo una aplicación consistente de las etiquetas XML definidas en el esquema de datos.

\subsection{Ingeniería de prompts estructural}

El \textit{prompt} enviado al modelo ha sido diseñado para actuar como una guía técnica de generación. Entre las directrices incorporadas destacan las siguientes:

\begin{itemize}
    \item Fidelidad textual: obligatoriedad de transcribir caracteres especiales, signos diacríticos y grafías propias de las lenguas mayas sin aplicar normalizaciones innecesarias.

    \item Jerarquía XML: instrucciones explícitas para encapsular la información siguiendo la estructura formal descrita en la DTD.

    \item Manejo de errores visuales: directrices sobre cómo proceder ante manchas de tinta, texto parcialmente oculto o imágenes borrosas, instando al modelo a emplear inferencia contextual para reconstruir palabras o segmentos incompletos.
\end{itemize}

\subsection{Limpieza de respuesta (Parsing)}

Dado que los LLMs suelen devolver sus respuestas encapsuladas en bloques de código Markdown mediante delimitadores de triple comilla invertida, \texttt{ai\_generator.py} incorpora una fase de limpieza o \textit{parsing} destinada a extraer exclusivamente el contenido XML generado.

Esta operación constituye un paso previo imprescindible para que la salida pueda ser procesada correctamente por los módulos de corrección y validación estructural, evitando que elementos de presentación ajenos al documento interfieran con el análisis sintáctico posterior.



\section{Estructuración, corrección y validación XML}

Uno de los mayores desafíos de emplear LLMs para la generación de datos es su naturaleza estocástica, que puede derivar en errores sintácticos de marcado. Para mitigar este riesgo, el sistema incorpora una capa de garantía de calidad (\textit{Quality Assurance}, QA) que combina corrección heurística y validación formal.

\subsection{Definición de la gramática de datos (DTD)}

La estructura jerárquica de las gramáticas mayas se ha formalizado en el archivo \texttt{validator/gramatica.dtd}. Este documento actúa como el contrato estructural que debe cumplir cualquier salida generada por el sistema.

El DTD establece las relaciones jerárquicas entre los distintos elementos del documento, definiendo qué etiquetas pueden aparecer, en qué orden deben hacerlo y cuáles son sus restricciones estructurales. Gracias a este mecanismo, se garantiza que los documentos XML producidos por la Inteligencia Artificial mantengan una organización coherente y compatible con las fases posteriores de procesamiento.

\subsection{Corrección heurística automática (\texttt{xml\_corrector.py})}

Antes de someter la respuesta de la IA a una validación formal, el sistema invoca el módulo \texttt{xml\_corrector.py}. Este componente aplica una serie de algoritmos de limpieza destinados a subsanar los errores más frecuentes detectados durante la fase de prototipado.

Entre las operaciones realizadas destacan:

\begin{itemize}
    \item Cierre de etiquetas: detección y clausura automática de etiquetas que el modelo ha dejado abiertas debido a límites de generación o errores de inferencia.

    \item Eliminación de redundancias: supresión de cabeceras XML duplicadas, bloques explicativos ajenos al documento o fragmentos de texto generados fuera de la estructura esperada.

    \item Normalización estructural: verificación de la existencia de un único elemento raíz, corrigiendo situaciones en las que el modelo produce múltiples nodos principales incompatibles con la sintaxis XML.
\end{itemize}

Estas operaciones permiten aumentar significativamente la tasa de éxito de las validaciones posteriores sin alterar el contenido semántico generado por el modelo.

\subsection{Validación de integridad estructural (\texttt{xml\_validator.py})}

Una vez corregido, el documento XML es procesado por el módulo de validación. Utilizando la librería \texttt{lxml}, el sistema compara la estructura generada con las reglas definidas en el archivo DTD.

El resultado de este proceso es binario:

\begin{itemize}
    \item Si el documento cumple todas las restricciones gramaticales, se autoriza su almacenamiento y posterior conversión a \LaTeX.

    \item Si se detecta cualquier incumplimiento estructural, el error es capturado y registrado automáticamente en el sistema de \textit{logs}.
\end{itemize}

Esta capacidad de detección ha permitido cuantificar experimentalmente la estabilidad de las diferentes configuraciones de temperatura analizadas en el Capítulo~\ref{cap:resultados}. De este modo, una ejecución puede clasificarse como fallida incluso cuando el contenido textual es correcto, siempre que la estructura jerárquica no respete las reglas definidas por la gramática formal.

La incorporación de esta capa de validación transforma una herramienta de generación textual en un sistema de extracción de datos con mayores garantías estructurales. Esta validación asegura que el XML respete la gramática formal definida, aunque no implica necesariamente una fidelidad textual completa respecto al documento original; dicha fidelidad se evalúa de forma separada mediante las métricas experimentales.

\section{Motor de conversión y tipografía científica (\texttt{converter.py})}

La etapa final del sistema tiene como objetivo transformar los datos estructurados en XML en un documento de alta calidad tipográfica apto para su consulta, análisis y publicación académica. Para ello, se ha desarrollado el módulo \texttt{converter.py}, encargado de realizar la traducción semántica entre el modelo de datos XML y el lenguaje de composición tipográfica \LaTeX.

\subsection{Mapeo semántico XML--\LaTeX}

El convertidor implementa una lógica de transformación que asocia cada etiqueta XML con su representación correspondiente en \LaTeX, preservando la jerarquía definida en la gramática original.

Entre las transformaciones más relevantes se encuentran:

\begin{itemize}
    \item Estructura jerárquica: los elementos que representan títulos, encabezados y divisiones documentales se convierten dinámicamente en comandos como \verb|\chapter{}|, \verb|\section{}| o \verb|\subsection{}| según su nivel de profundidad.

    \item Cuerpo de texto: los bloques de contenido textual se normalizan para asegurar una correcta gestión de párrafos, saltos de línea y separación entre elementos, siguiendo los estándares de edición científica.
\end{itemize}

Este mecanismo garantiza que la organización lógica del documento permanezca inalterada durante todo el proceso de conversión.

\subsection{Tratamiento de glosas y caracteres especiales}

Uno de los principales retos técnicos abordados por este módulo es la correcta representación de los caracteres específicos de las lenguas mayas, incluyendo saltillos, vocales modificadas y otros símbolos lingüísticos presentes en las gramáticas normativas.

Para ello, el sistema incorpora las siguientes estrategias:

\begin{itemize}
    \item Codificación Unicode: toda la salida se genera en formato UTF-8, permitiendo que motores de composición modernos como XeLaTeX o LuaLaTeX representen correctamente los caracteres especiales.

    \item Formateo de glosas: los ejemplos lingüísticos se estructuran mediante tablas, listas y entornos especializados que preservan la alineación entre formas originales, segmentaciones morfológicas y traducciones, evitando los problemas de desorganización visual característicos de las conversiones automáticas desde texto plano.
\end{itemize}

Estas medidas resultan fundamentales para garantizar la fidelidad lingüística del documento final.

\subsection{Automatización de la compilación}

El proceso de conversión se encuentra completamente automatizado. Una vez que el archivo XML supera satisfactoriamente la validación estructural, \texttt{converter.py} genera de forma inmediata un archivo \texttt{.tex} completo y listo para compilar.

Entre las tareas realizadas automáticamente destacan:

\begin{itemize}
    \item Generación del preámbulo técnico necesario para la configuración del documento.

    \item Inclusión de los paquetes requeridos para el soporte de idiomas, codificación y maquetación.

    \item Construcción automática de la estructura jerárquica utilizada posteriormente por \LaTeX{} para generar índices y referencias cruzadas.

    \item Producción de un archivo final preparado para su compilación directa a PDF sin necesidad de intervención manual.
\end{itemize}

Esta capacidad de automatización completa el ciclo de digitalización propuesto en este trabajo: la información que originalmente se encuentra en documentos PDF procedentes de fotografías o digitalizaciones de obras impresas se transforma en un recurso digital estructurado, consultable, reutilizable y presentado con una calidad tipográfica equivalente a la de una edición académica profesional.






\section{Automatización de experimentos y orquestación (\texttt{experiment\_runner.py})}

Para llevar a cabo un estudio empírico sobre el comportamiento de los LLMs en la digitalización lingüística, se ha desarrollado un módulo de orquestación denominado \texttt{experiment\_runner.py}. Este componente permite ejecutar pruebas masivas de forma sistemática, eliminando el sesgo asociado al procesamiento manual y garantizando la reproducibilidad de los experimentos.

\subsection{Diseño del barrido de hiperparámetros}

El módulo está diseñado para realizar experimentos sistemáticos sobre distintas configuraciones del sistema. En este trabajo se han llevado a cabo dos experimentos principales: un barrido sobre el parámetro de temperatura del modelo y una comparativa entre diferentes modelos de la familia Gemini.

En ambos experimentos, el eje de análisis vertical estuvo formado por las páginas del \textit{test set}. En el experimento de temperatura, el eje horizontal correspondió a los valores \(T \in \{0.0,\;0.1,\;0.25,\;0.5,\;0.75,\;1.0,\;1.5,\;2.0\}\). En el experimento de modelos, el eje horizontal correspondió a las distintas arquitecturas Gemini evaluadas. Para obtener el valor de cada celda de las tablas de resultados, el programa se ejecutó tres veces bajo la misma configuración y se calculó la media aritmética de las tres salidas.

Esta estrategia permite reducir parcialmente el efecto de la variabilidad propia de los modelos generativos y obtener una estimación media para cada combinación de página y configuración. No obstante, al no conservarse los valores individuales de cada una de las tres ejecuciones, el análisis se limita a las medias obtenidas.

\subsection{Flujo de orquestación modular}

El módulo \texttt{experiment\_runner.py} actúa como coordinador del sistema, invocando secuencialmente los distintos componentes del \textit{pipeline} para cada experimento. Este flujo de evaluación se resume en la Figura~\ref{fig:uml_evaluacion_experimentos}.
El flujo de ejecución sigue el siguiente orden:

\begin{enumerate}
    \item Obtención del documento de prueba mediante \texttt{pdf\_processor.py}, generando el fragmento PDF segmentado en bytes correspondiente al rango de páginas evaluado.

    \item Ejecución de \texttt{ai\_generator.py} utilizando la temperatura correspondiente a la iteración experimental.

    \item Aplicación de las reglas de corrección implementadas en \texttt{xml\_corrector.py}.

    \item Validación formal del resultado mediante \texttt{xml\_validator.py}.

    \item Conversión del XML validado a \LaTeX{} utilizando \texttt{converter.py}.
\end{enumerate}

La Figura~\ref{fig:uml_evaluacion_experimentos} muestra el flujo de evaluación empleado para comparar la salida generada con el XML de referencia. Este proceso permite calcular la tasa de error y derivar el porcentaje de similitud utilizado posteriormente en el análisis experimental.

\subsection{Persistencia y versionado de resultados}

Una de las características más relevantes de este módulo es su sistema de almacenamiento jerárquico de resultados. Cada ejecución se organiza automáticamente según criterios como el modelo empleado, el idioma procesado y la temperatura utilizada.

El sistema incorpora además un mecanismo de versionado incremental. Cada resultado generado se almacena con un identificador de versión secuencial (\texttt{v1}, \texttt{v2}, \texttt{v3}, etc.), lo que permite mantener la trazabilidad de las ejecuciones realizadas.

En este trabajo, las ejecuciones repetidas se han utilizado para obtener una media por celda experimental. Sin embargo, dado que no se han conservado los valores individuales de cada intento, no se realiza un análisis estadístico de la variabilidad entre repeticiones. Por tanto, las conclusiones deben interpretarse como un análisis empírico exploratorio basado en medias agregadas.

Adicionalmente, el orquestador registra información relativa al tiempo de respuesta de la API y monitoriza posibles excepciones durante la ejecución. Entre ellas se incluyen errores de conectividad, interrupciones del servicio o agotamiento de cuotas de uso. La captura y almacenamiento de estos eventos permite reanudar experimentos interrumpidos sin comprometer la integridad de los datos obtenidos.

Gracias a este nivel de automatización, el sistema trasciende el ámbito de una prueba conceptual y se convierte en una infraestructura experimental capaz de generar evidencia empírica sobre la fiabilidad de los Modelos de Lenguaje de Gran Tamaño en tareas de preservación y digitalización del patrimonio lingüístico. Los datos producidos por este módulo constituyen la base cuantitativa empleada posteriormente en el análisis de resultados presentado en el Capítulo~\ref{cap:resultados}.
